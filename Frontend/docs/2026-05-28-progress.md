## 현재 목표

Android 상담 앱에서 온디바이스 Gemma 4 E4B IT 모델을 실행하고, 사용자의 이미지/음성/헬스 데이터/대화 기록/중요 기억을 상담 맥락으로 활용할 수 있는 구조를 만든다.

0527 문서 기준으로 다음 항목이 진행 중이었다.

- 이미지/음성 입력 중간 구현
- 상담 세션 JSON 내보내기/불러오기 중간 구현
- 헬스 데이터 프롬프트 주입 예정
- 작업 시작 시 빌드 검증 필요

처음 `.\gradlew.bat :app:assembleDebug` 실행 시 셸이 Java 8을 사용해 Android Gradle Plugin 8.12.3 요구 조건을 만족하지 못했다. Android Studio 내장 JBR을 `JAVA_HOME`으로 지정해 다시 빌드했고, 이후 실제 Kotlin 컴파일 오류를 확인했다.

## 이미지/음성 입력

완료한 내용:

- Android Photo Picker로 이미지 선택
- `GetContent("audio/*")`로 음성 파일 선택
- `RECORD_AUDIO` 권한을 받아 앱 안에서 즉석 녹음
- `녹음` 버튼으로 녹음 시작, `정지`로 녹음 종료 후 첨부
- 선택한 URI를 앱 내부 `files/chat_inputs` 디렉터리로 복사
- 즉석 녹음 파일도 앱 내부 `files/chat_inputs` 디렉터리에 저장
- LiteRT-LM `miniaudio decoder`가 `.m4a`를 읽지 못해 즉석 녹음은 `16kHz mono PCM WAV`로 변경
- 기존 음성 파일 선택은 안전성을 위해 WAV PCM 파일만 허용
- m4a/mp3/aac 자동 변환은 일부 기기 코덱에서 앱 프로세스가 종료될 수 있어 제거
- 오디오 직접 첨부는 60초 이하로 제한
- Photo Picker 이미지는 LiteRT-LM 네이티브 디코더 안정성을 위해 앱 내부에서 768px 이하 JPEG로 변환 후 첨부
- 이미지 변환은 원본 전체 디코딩으로 인한 종료를 줄이기 위해 Android 28+에서 `ImageDecoder` target size를 사용
- 이미지/오디오 첨부 포함 LiteRT-LM 호출이 Kotlin 예외로 실패하면 같은 메시지를 텍스트-only로 한 번 더 보내는 fallback 추가
- 복사 중 실패 파일이 남지 않도록 `.partial` 파일에 먼저 저장한 뒤 최종 파일로 교체
- 이미지 첨부는 `ChatMessage.imagePath`로 저장
- 음성 첨부는 `ChatMessage.audioPath`로 저장
- 말풍선에 첨부 라벨 표시
- LiteRT-LM 전달 시:
  - `Content.ImageFile(imagePath)`
  - `Content.AudioFile(audioPath)`
- 첨부 실패 시 앱이 죽지 않고 상태 메시지로 표시

주의:

- LiteRT-LM 멀티모달 모델이 실제 기기에서 이미지/오디오 입력을 정상 처리하는지는 별도 기기 테스트가 필요하다.
- `말하기` 버튼은 Android 음성 인식을 사용해 텍스트로 변환하는 경로이고, `녹음`/`음성파일`은 모델에 오디오 파일을 직접 전달하는 경로다.
- 사용자가 보고한 `Failed to initialize miniaudio decoder, error code: -10`은 miniaudio가 파일을 유효한 오디오로 디코딩하지 못했다는 의미로 보고, WAV PCM 경로로 우회했다.

## 채팅 입력/키보드

- `AndroidManifest.xml`의 `windowSoftInputMode="adjustResize"`는 유지한다.
- 일부 삼성 키보드/기기에서 `imePadding()`만으로는 입력창이 올라오지 않아, Compose `WindowInsets.ime`에서 실제 키보드 하단 높이를 읽고 입력창 하단 패딩에 직접 더하도록 변경했다.
- 이 방식은 사용자별 키보드 높이 차이를 따라가도록 하기 위한 임시 안정화 조치다.

## 테마/화면 분위기

- 기존 앱은 시스템 다크 테마와 Material dynamic color를 따라가면서 화면이 너무 어둡고 칙칙하게 보일 수 있었다.

- 테마 선택값은 `SharedPreferences` 기반 `AppSettingsStore`에 저장한다.
- Material dynamic color는 예측하기 어려운 색감 때문에 우선 끄고, 앱 자체 색상 팔레트를 사용한다.
- 상단 앱바의 테마 버튼에서 테마를 바꿀 수 있다.
- 기존 보라색 중심 팔레트 대신 밝은 배경과 파랑/초록/코랄 계열 포인트 색을 사용한다.

## 사고 모드

- Gemma 공식 문서 기준으로 Gemma instruction-tuned 모델은 `user`/`model` 두 역할 중심으로 동작하고, 별도 system role 대신 첫 사용자 프롬프트에 시스템 수준 지시를 넣는 방식이 권장된다.
- Gemma 문서와 모델 카드에서는 reasoning 작업에 적합하다고 설명하지만, 별도 `thinking` 전용 API나 role을 확인하지는 못했다.
- 따라서 앱에서는 프롬프트 지시로 “사고 모드”를 구현한다.
- 완료한 내용:
  - 채팅 입력 영역에 `사고 자동`/`사고 켬`/`사고 끔` 순환 버튼 추가
  - `자동` 모드는 위험 신호, 복잡한 상담 판단, 첨부, 중요 기억, 긴 메시지, 건강/관계/패턴 관련 표현이 있을 때 사고 모드 적용
  - `켜짐` 모드는 사용자가 직접 해당 메시지에 사고 모드 적용
  - `꺼짐` 모드는 짧고 일반적인 응답 우선
  - 사고 모드 적용 중에는 UI에 `사고 중...` 표시
  - 모델이 긴 사고 과정을 그대로 노출하지 않고, 핵심 판단 근거와 답변만 간결히 제시하도록 지시

## 중요 기억

완료한 내용:

- 사용자가 다음과 같은 표현을 쓰면 해당 메시지를 중요 기억으로 저장한다.
  - `기억해`
  - `기억해줘`
  - `기억해 줘`
  - `저장해`
  - `저장해줘`
  - `저장해 줘`
  - `중요해`
  - `중요한`
  - `잊지마`
  - `잊지 마`
- 중복 저장은 피한다.
- 저장 시 상태 메시지로 `중요한 내용으로 저장했습니다.`를 표시한다.
- 중요 기억은 시스템 프롬프트와 현재 메시지 프롬프트에 함께 주입된다.

기본 중요 기억:

- 상담 응답은 진단이나 치료를 대신하지 않고, 감정 반영과 정리, 작은 다음 행동 선택을 돕는 방식으로 유지한다.
- 불안, 긴장, 감정 과부하가 있을 때는 느린 호흡, 몸 감각 알아차리기, 5-4-3-2-1 감각 그라운딩, 짧은 산책, 점진적 근육 이완 같은 부담 낮은 안정화 방법을 선택지로 제안한다.
- 자살, 자해, 타해, 학대, 폭력, 응급 위험 신호가 있으면 일반 조언보다 안전 확인과 긴급 지원 안내를 우선한다.

중요 기억 UI:

- 채팅 화면 상단에 `기억` 버튼 추가
- 기본 기억과 사용자 기억을 함께 표시
- 기본 기억은 삭제 버튼을 표시하지 않음
- 사용자 기억만 삭제 가능
- 사용자 기억 삭제 시 세션 저장 파일과 모델 시스템 프롬프트를 갱신

## 마음챙김/안정화 지침

기본 시스템 프롬프트에 다음 방향을 추가했다.

- 마음챙김은 감정을 없애는 방법이 아니라 지금의 생각, 감정, 몸 감각을 판단하지 않고 알아차리는 연습으로 설명
- 불안, 압박감, 긴장, 감정 과부하가 있을 때 짧고 부담 낮은 안정화 연습을 선택지로 제안
- 예시:
  - 느린 호흡
  - 몸 감각 알아차리기
  - 5-4-3-2-1 감각 그라운딩
  - 짧은 산책
  - 점진적 근육 이완
  - 물 마시기
- 트라우마, 공황, 해리, 극심한 불안, 자해 충동이 있으면 눈을 감거나 내면에 오래 머무는 연습을 강요하지 않고 외부 지향 안정화부터 제안
- 증상이 지속되거나 일상 기능을 크게 방해하면 정신건강 전문가나 의료기관 상담을 권함

참고 기준:

- NIH/NCCIH 스트레스와 마음챙김 자료
- NCCIH 명상/마음챙김 개요
- NIMH 심리치료와 coping/mindfulness 설명

## RAG-lite / 과거 대화 검색

완료한 내용:

- `ChatMemoryStore` 추가
- Android 내장 SQLite를 사용해 `chat_memory.db` 생성
- `chat_messages` 테이블에 메시지 저장
- `chat_messages_fts` FTS4 가상 테이블 생성
- 현재 세션 메시지를 인덱싱
- 앱 시작/세션 불러오기/메시지 저장 후 인덱스 갱신
- 새 사용자 메시지 전송 시:
  - 현재 입력으로 과거 관련 메시지 검색
  - FTS 검색 우선
  - 결과가 없으면 `LIKE` fallback
  - 방금 보낸 현재 메시지는 검색 결과에서 제외
  - 최대 4개 과거 대화를 `[현재 메시지와 관련 있을 수 있는 과거 대화]` 블록으로 프롬프트에 주입

현재 RAG-lite 판단:

- 벡터 임베딩 없이 SQLite FTS로 시작했다.
- 온디바이스 앱에서는 의존성과 빌드 안정성을 고려할 때 1차 검색 메모리로 충분히 실용적이다.
- 향후 필요하면 Room + FTS 또는 임베딩 기반 벡터 검색으로 확장 가능하다.
